## 2. ЭТАП Б — ИНТЕГРАЦИЯ ПЛАТЕЖЕЙ ЧЕРЕЗ PRODAMUS (2-3 дня)

### 2.1. Что нужно от пользователя (без меня нельзя)
- ✅ Самозанятость открыта
- 📋 Зарегаться на https://prodamus.ru:
  1. Email + телефон (тот же что в боте)
  2. Статус: **Самозанятый (НПД)**
  3. Налогообложение: ИНН + автоподача чеков в «Мой Налог»
  4. Категория: «Программное обеспечение / SaaS-сервис»
- 📋 В кабинете Prodamus → **Создать продукт**:
  - 4 продукта (по тарифам): «PRO ИП 290 ₽/мес», «PRO Бух 990 ₽/мес», «MAX 1990 ₽/мес», + «Триал 1 день — 0 ₽»
  - Тип оплаты: **Подписка (ежемесячно)**
  - Auto-prolongation: ON
- 📋 Получить из Prodamus:
  - **API token** (для серверных запросов)
  - **Secret key** (для проверки подписи webhook)
  - **URL формы** для каждого продукта (`https://payment.prodamus.ru/{shopname}/form/...`)

### 2.2. Архитектура интеграции
```
[ Лендинг / Бот ]
       ↓ click "Купить PRO"
[ Vercel Edge Function /api/checkout ]
       ↓ генерит подписанный URL Prodamus с user_id и tariff
[ Prodamus payment page ]
       ↓ user платит
[ Prodamus webhook → /api/prodamus-webhook ]
       ↓ проверка HMAC подписи
       ↓ запись в БД (Railway PostgreSQL) tier=PRO, активен_до=+1мес
       ↓ Telegram-уведомление в бот: «✅ оплата прошла, активирую тариф»
[ Telegram-бот ] получает signal → апдейтит UI
```

### 2.3. Файлы для создания
| Файл | Назначение |
|---|---|
| `api/checkout.js` | Edge Function: получает user_id+tariff, возвращает Prodamus payment URL |
| `api/prodamus-webhook.js` | Edge Function: webhook receiver, проверка HMAC, апдейт БД, нотификация бота |
| `lib/prodamus.js` | HMAC sign/verify, format payment URL |
| `.env.example` | Документация переменных (PRODAMUS_TOKEN, PRODAMUS_SECRET, BOT_TOKEN, RAILWAY_DATABASE_URL) |
| `.vercel/project.json` | env-bindings (через `vercel env add`) |

### 2.4. Тесты безопасности
- HMAC-проверка на webhook ОБЯЗАТЕЛЬНА — без неё подделать оплату легко
- Idempotency: сохраняем `payment_id` в БД и игнорируем повторные webhook
- Rate-limit на /api/checkout (max 5 req/min с одного IP)

### 2.5. Что показываем на лендинге
- На pricing-секции — кнопки **«Оформить подписку»** ведут на Prodamus
- Триал 1 день — после регистрации в боте автоматом, без оплаты
- При оплате с лендинга — Prodamus просит email (= идентификатор), позже user привязывает к боту через `/connect код123`

## 4. ЭТАП Г — РАСШИРЕННЫЙ КАЛЬКУЛЯТОР (1-2 дня)

### 4.1. Текущие ограничения
- Light-расчёт квартальный (× 4 на год)
- Без региональных ставок (только стандарт)
- Без переноса убытков, ОС, мин.налога 1%
- Результат не сохраняется

### 4.2. Расширение
**Новые поля:**
- ☑ Регион (dropdown: Москва, СПб, ХМАО, Татарстан, ... + «другой → ставка X%»)
- ☑ Период расчёта: квартал | полугодие | 9 месяцев | год
- ☑ Уже уплаченные авансы (для расчёта «к доплате»)
- ☑ ОС-списание за период (для УСН Д−Р)
- ☑ Перенос убытков прошлых лет (для УСН Д−Р, только в годовом)
- ☑ Прогноз годового дохода → НДС-предупреждение (>20 млн)

**Новый блок результата:**
- Налог по периодам (Q1 / H1 / 9M / Year) — таблица
- График нарастающим итогом
- «К оплате до 28 числа следующего месяца»
- «Если ты пропустишь срок — пени = X% от Y руб. = Z руб./день»

**Сравнение режимов (фишка для лидогенерации):**
- Кнопка **«Сравнить с УСН Д−Р»** автоматом считает альтернативу
- Если выгоднее другой режим — большая жёлтая плашка «Тебе выгоднее перейти на ___ → сэкономишь N ₽»
- CTA: «Хочешь смену режима с правильным заявлением? → Telegram-бот»

**Сохранение результата:**
- Кнопка **«Прислать на email»** → Formsubmit + body содержит весь расчёт
- Кнопка **«Сохранить как PDF»** → клиентская генерация через jsPDF (без бэкенда)
- В подвале PDF: «Расчёт от AI-Бухгалтер.com, НК РФ 2026, дата X»
